Interactive Tutorial: Speculative Execution and Cache-Based Leakage

This is a toy teaching model of how a CPU can transiently execute a mispredicted path, touch data-dependent cache lines, and leave behind a timing signal. It is designed for classroom explanation, not for attacking real systems.

array1 → array2 mapping

This tab focuses only on the lookup y = array2[array1[x] * 256]. The goal is to show how a byte loaded from array1 selects one widely separated region of array2.

In this version, the spacing is fixed at 128 bytes and the number of highlighted candidate targets is fixed at 16. This makes it easier to see the standard teaching pattern where one of 16 possible values selects one of 16 widely separated regions.
Pick an index x. The byte array1[x] becomes the selector. Multiplying by 128 spaces the candidate array2 accesses far apart so each value tends to land in a different cache line or cache-line group.

What this tutorial is showing

The key idea is that the architectural result of the mispredicted path is thrown away, but microarchitectural side effects can remain. A cache line loaded during transient execution may still be present afterward, and that presence can be detected by timing.

Classroom framing

A useful way to explain this is: the CPU guesses, touches memory before it knows whether that work was legitimate, and then later says “never mind” architecturally — but the cache was still changed. That cache change becomes the signal.

Important caveats

Real processors are much more complex than this toy visualization. Real attacks depend on the exact microarchitecture, timer availability, mitigation state, and many other details. This page intentionally stays abstract and focuses on the learning model.
